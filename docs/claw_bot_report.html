What This Report Covers

Workflow: When new commits are pushed to Formelife/Android-Member-App/feature/next/studio-app, claw_bot_studio.py is triggered. It reads the changed Kotlin files, diffs them against the last commit, calls Claude to generate or update test scripts, and opens a PR against jahnaviyeddanapudy/qa_forme_automation/main under surfaces/studio_android/.

Observed baseline (June 2026): 50 commits · 71 unique Kotlin files · 4 parallel batches · 4 PRs created → 4 merged. Pricing: claude-sonnet-4-6 — $3.00 / M input tokens · $15.00 / M output tokens.

One-time catch-up cost (not part of recurring projections): The initial sweep of 71 backlogged Kotlin files cost ~$1.55 (parallel, 4 batches) or ~$1.24 (single branch, high truncation risk). This is a one-time event when the bot is first set up or after a long gap. Recurring runs (daily/weekly) will be significantly cheaper because they only process files changed since the last run.

Approach Overview

⚡ Parallel Feature Branches

  • Faster wall-clock — batches run simultaneously
  • Isolated PRs per feature area — easier to review
  • One batch failure does not block others
  • Smaller Claude context per call → no truncation, higher quality
  • Branch name collisions without GITHUB_RUN_ID suffix (already fixed)
  • Multiple PRs to review and merge per cycle
  • Higher total input cost: QA context re-read once per batch
  • More workflow orchestration complexity

⇄ Single Branch

  • One PR — simpler review and merge
  • Lower input overhead: QA context loaded once per run
  • Simpler workflow, no batch coordination
  • Best for daily push-triggered runs (few files per push)
  • Slower wall-clock for large batches
  • JSON truncation risk when output hits MAX_TOKENS
  • One failure aborts the entire run
  • Large mixed context may reduce output quality across feature areas

Impact: Costs are not flat across the quarter. Runs in week 13 will cost more than runs in week 1 because the QA library is larger. The recurring projections below use the average context size over the quarter.

Assumptions & Caveats

Model: claude-sonnet-4-6 · Input $3.00/M tokens · Output $15.00/M tokens

Observed change rate: 50 commits/quarter touching studio-app Kotlin · 71 unique files in backlog · ~6 files/week average going forward

Push frequency assumption: ~40% of working days have at least one Kotlin commit (36/90 days). If developers push multiple commits per day touching different files, cost scales proportionally — e.g. 3 pushes/day on active days → up to 108 triggered runs → cost could reach ~$13/quarter on daily cadence.

Context growth: Assumes ~1 new page file + ~1 new test file generated per week (conservative). If the bot is more productive (e.g. 3–4 new files/week), end-of-quarter context could reach ~35,000 tokens, adding another ~$0.50–$1.00 to quarterly cost.

Per-file token basis: ~4,500 input tokens per Kotlin file (3,000 char source + diff) · ~12,500–22,000 tokens QA context (grows over quarter) · ~2,000–4,000 output tokens per call

Not included: GitHub Actions runner minutes (free tier covers this) · Human review time · member_android bot (separate workflow, similar cost profile)

Recommendation

Daily (push-triggered)

Single branch. Batches are small (2–4 files per push). No truncation risk. One PR per push is clean to review. Parallel adds complexity with no benefit at this scale. Recurring est.: ~$4.46/quarter. First quarter (with catch-up): ~$5.70.

Weekly (scheduled or catch-up)

Single branch for ≤10 files. Switch to parallel when batch exceeds ~15 files (post-freeze, sprint-end, or quarterly sweep). Recurring est.: ~$2.73/quarter. First quarter (with catch-up): ~$3.97.

Hybrid (recommended)

Daily push-triggered single-branch + weekly parallel sweep as backstop. Studio + member bots combined: ~$11–14/quarter recurring (after initial catch-up). Both bots first-quarter total: ~$13–17.

Key watch-outs

Parallel runs must use GITHUB_RUN_ID in branch names (already in place). MAX_TOKENS ≥ 16,000 to avoid JSON truncation (already applied). Mirror all fixes to claw_bot_member.py / member_test_sync.yml before running a large member backlog.
