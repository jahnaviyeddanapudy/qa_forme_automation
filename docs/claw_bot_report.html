What This Report Covers

Workflow: When new commits are pushed to Formelife/Android-Member-App/feature/next/studio-app, claw_bot_studio.py reads the changed Kotlin files, diffs them, calls Claude (claude-sonnet-4-6) to generate or update test scripts, and opens a PR against jahnaviyeddanapudy/qa_forme_automation/main under surfaces/studio_android/.

Observed baseline (June 2026): 50 commits · 71 unique Kotlin files · 4 parallel batches · 4 PRs created → merged. Pricing: $3.00 / M input tokens · $15.00 / M output tokens.

One-time catch-up cost (not part of recurring projections): The initial sweep of 71 backlogged files cost ~$1.55 (parallel) or ~$1.24 (single, truncation risk). Recurring runs process only files changed since the last run and are significantly cheaper.

Approach Overview

⚡ Parallel Feature Branches

  • Faster wall-clock — batches run simultaneously
  • Isolated PRs per feature area — easier to review
  • One batch failure does not block others
  • Smaller Claude context per call → no truncation, higher quality
  • Branch collisions without GITHUB_RUN_ID suffix (already fixed)
  • Multiple PRs to review and merge per cycle
  • QA context re-read once per batch — cost grows sharply with test suite size
  • More workflow orchestration complexity

⇄ Single Branch

  • One PR — simpler review and merge
  • QA context loaded once per run — cost scales better at large test suite
  • Simpler workflow, no batch coordination
  • Strongly preferred when test suite exceeds ~200 cases
  • Slower wall-clock for large batches of changed files
  • JSON truncation risk when output hits MAX_TOKENS
  • One failure aborts the entire run

Scale Projection: 800 Test Cases

What 800 test cases means for context: Assuming avg 8 test cases per test file → ~100 test files. Plus ~80 page objects. Context size: 80 pages × 1,500 chars + 100 tests × 2,000 chars = 320,000 chars ≈ ~80,000 tokens per run.

Key finding: At 800 test cases, the QA context (~80,000 tokens, ~$0.24/run) completely dwarfs the source file tokens (~13,500 tokens for 3 files, ~$0.04/run). Parallel batching multiplies this 80,000-token context cost per batch — making single branch strongly preferred at this scale.

Context window warning: 80,000 tokens of QA context + source files + system prompt ≈ 95,000–115,000 tokens input. This is within Claude’s 200k context window but getting large. If context exceeds the window, the bot would need to implement selective context loading (only retrieve test files relevant to the changed Kotlin files) rather than loading all 800 test cases every run.

Assumptions & Caveats

Model: claude-sonnet-4-6 · Input $3.00/M tokens · Output $15.00/M tokens

Observed change rate: 50 commits/quarter · 71 unique files in backlog · ~6 files/week average going forward

Push frequency: ~40% of working days have at least one Kotlin commit (36/90 days). Multiple commits per day scale cost proportionally — e.g. 3 pushes/day on active days → ~108 triggered runs → up to ~$33/quarter at 800 test cases, daily.

800 test cases context estimate: 80 page files × 1,500 chars + 100 test files × 2,000 chars ≈ 80,000 tokens. Actual size depends on file verbosity.

Context window limit: At 800 test cases, input per run (≈ 95k–115k tokens) stays within Claude’s 200k context window. Beyond ~1,500 test cases, selective context loading would be required.

Not included: GitHub Actions runner minutes (free tier) · Human review time · member_android bot (similar cost profile)

Recommendation

Now (<200 test cases)

Single branch for daily pushes. Parallel for weekly when batch >15 files. Q1 total (with catch-up): ~$4–6. Studio + member combined: ~$8–12.

At 800 test cases

Single branch always. Context cost makes parallel 41–70% more expensive. Weekly single is the best tradeoff: ~$6.42 Q1. Consider selective context loading to keep cost flat as suite grows further.

Hybrid (recommended near-term)

Daily push-triggered single branch + weekly sweep as backstop. Gives immediate coverage without large batches. Revisit selective context loading when test suite exceeds ~400 cases.

Key watch-outs

Parallel runs require GITHUB_RUN_ID suffix (already in place). MAX_TOKENS ≥ 16,000 to avoid JSON truncation (already applied). Mirror fixes to claw_bot_member.py before large member backlog.
