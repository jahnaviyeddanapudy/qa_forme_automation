What This Report Covers

Workflow: When new commits are pushed to Formelife/Android-Member-App/feature/next/studio-app, claw_bot_studio.py reads the changed Kotlin files, diffs them, calls Claude (claude-sonnet-4-6) to generate or update test scripts, and opens a PR against jahnaviyeddanapudy/qa_forme_automation/main under surfaces/studio_android/.

Observed baseline (June 2026): 50 commits · 71 unique Kotlin files · 4 parallel batches · 4 PRs merged. Pricing: $3.00 / M input tokens · $15.00 / M output tokens.

One-time catch-up cost (excluded from recurring projections): Initial sweep of 71 backlogged files cost ~$1.55 (parallel) or ~$1.24 (single, truncation risk). Recurring runs process only files changed since the last run.

Approach Overview

⚡ Parallel Feature Branches

  • Faster wall-clock — batches run simultaneously
  • Isolated PRs per feature area — easier to review
  • One batch failure does not block others
  • Smaller Claude context per call → no truncation at current scale
  • Branch collisions without GITHUB_RUN_ID suffix (already fixed)
  • Multiple PRs to review and merge per cycle
  • QA context re-read per batch — cost multiplies sharply with test suite size
  • More workflow orchestration complexity

⇄ Single Branch

  • One PR — simpler review and merge
  • QA context loaded once per run — scales much better at large test suite
  • Simpler workflow, no batch coordination
  • Strongly preferred at 800+ test cases
  • Slower wall-clock for large batches of changed files
  • JSON truncation risk when output hits MAX_TOKENS
  • One failure aborts the entire run

Assumptions & Caveats

Model: claude-sonnet-4-6 · Input $3.00/M tokens · Output $15.00/M tokens

Observed change rate: 50 commits/quarter · 71 unique files in backlog · ~6 files/week average going forward

Push frequency: ~40% of working days have at least one Kotlin commit (36/90 days). Multiple commits/day scale cost proportionally.

800 test cases: 80 page files × 1,500 chars + 100 test files × 2,000 chars ≈ 80,000 tokens context.

1,500 test cases: 150 page files × 1,500 chars + 188 test files × 2,000 chars ≈ 150,000 tokens context. Total input per run ~164,000 tokens — fits within 200k window but tight for large weekly batches. Selective context loading (only load files relevant to the changed Kotlin feature) is required beyond this scale.

Not included in Claude API cost: GitHub Actions runner minutes (see section below) · Human review time · member_android bot (similar cost profile)

GitHub Actions Runner Minutes

What are runner minutes? Every time the claw bot workflow runs, GitHub spins up a virtual machine (the runner) to execute the job — checkout, pip install, Python script, PR creation. The time that VM is alive is billed as runner minutes.

Who gets charged? Minutes are billed to the repo where the workflow lives. The claw bot runs on Formelife/Android-Member-App, so minutes are charged to Formelife's GitHub account, not to jahnaviyeddanapudy.

Public vs private: Public repos get unlimited free minutes. Minutes only count against the free tier for private repos.

Recommendation by Scale

Now — ~50 test cases

Single branch daily (push-triggered). Parallel optional for weekly when batch >15 files. Q1 total: ~$4–6.

800 test cases

Single branch always. Parallel is 41–70% more expensive due to context multiplication. Weekly single: ~$6.42 Q1. Avoid daily cadence if possible (~$12.67 Q1).

1,500 test cases

Selective context loading required. Loading 150k tokens every run hits the 200k window limit for larger batches. Bot must learn to load only the relevant page/test files per changed Kotlin feature. Weekly single best until then: ~$9.51 Q1.

Key watch-outs

Parallel runs require GITHUB_RUN_ID suffix (in place). MAX_TOKENS ≥ 16,000 to avoid truncation (applied). Mirror fixes to claw_bot_member.py before large member backlog.
